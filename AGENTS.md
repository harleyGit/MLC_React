# AGENTS.md

## 入口
- 先读本文件，再读对应模块规则。

## 最高优先级通用规则
- 只在当前 workspace 内工作。
- 先看上下文、调用链、相似实现，再改代码。
- 最小必要改动；不做无关重构。
- 优先复用现有实现；保持现有行为稳定。
- 冲突时优先：更保守、更兼容、更小改动。
- 发现架构或实现不合适时，先用最少文字说明问题、建议、收益/风险；获同意后再改。
- 优先采用主流稳定架构、通行工程实践和常用优秀设计模式。
- 不擅自修改公共接口、对外字段、错误语义、响应结构。
- 不擅自新增第三方依赖。
- 新增或修改方法、函数、关键变量时，补简洁注释，写清职责与关键约束。
- 修改后必须自检；有条件就编译、测试、lint；未执行不得说通过。
- 不把缓存、日志、构建产物、临时文件留在工程目录。

## 优先级
1. 用户当前要求
2. 本文件“最高优先级通用规则”
3. 本文件其他规则
4. `rules/common.md`
5. 对应工程类型规则
6. 项目现有实现

## 模块加载
所有任务都加载：
- `rules/common.md`

按工程类型追加：
- React：`rules/react.md`
- iOS：`rules/ios.md`
- Go：`rules/go.md`

## 默认流程
1. 读任务
2. 读本文件
3. 读 `rules/common.md` 和对应类型规则
4. 看上下文和相似实现
5. 按最小必要改动实施
6. 自检，并尽量编译 / 测试 / lint
7. 真实说明改动、原因、影响、验证结果

## React 工程补充
- 保持现有组件结构、路由、状态流和接口调用方式一致。
- 不为“更优雅”改变业务逻辑、props 契约、接口字段或页面交互。
- 先修正确性，再做结构优化和性能优化。
- 前端请求层、签名、鉴权、错误处理改动时，必须与后端接口行为对齐后再落代码。

## 输出要求
- 说明真实修改的文件。
- 说明改动内容、原因、潜在影响。
- 明确写出实际执行的编译、测试、lint 结果；未执行写原因。
